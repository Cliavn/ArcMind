# 质量门禁

本文件定义 agent 修改代码后的最低完成标准。

## 代码质量

- 保持修改范围最小，避免顺手重构无关代码。
- 优先复用现有目录结构、命名风格、接口和工具链。
- 新抽象必须降低真实复杂度，不能只为了显得整洁。
- 公共接口、数据结构、配置项和错误语义必须清晰。
- 错误处理应可观察、可恢复或可明确失败，不吞掉异常。
- 流式模型请求、音频录制、播放、WebGL 渲染和本地存储必须有明确生命周期。
- 不把 API Key、token、私密对话、语音样本、本地隐私路径写入代码、测试或文档。

## 测试标准

- 新增业务逻辑应优先补充单元测试。
- 修复 bug 应优先补充回归测试，证明问题不会复现。
- 跨模块流程应补充集成测试或端到端验证。
- UI 和视觉改动应补充截图检查或手动验收记录。
- WebGL/Three.js 场景必须验证非空白、尺寸正确、状态切换可见。
- 无法添加测试时，必须说明原因、手动验证方式和剩余风险。

## 验证标准

agent 完成代码修改前应尽量执行：

- 与本次修改最相关的测试。
- 类型检查或编译检查。
- lint 或格式检查。
- 必要的手动验证。
- 涉及视觉时进行桌面和窄窗口截图检查。
- 评估运行中的项目是否需要重启，并在需要时重启或说明无法重启原因。

如果项目尚未定义命令，agent 应先检查技术栈文件、包管理器和现有脚本；仍不存在时，在最终报告中说明“项目尚未建立验证命令”。

## 文档同步标准

以下情况必须同步更新 `docs/agent/`：

- 新增或删除模块。
- 改变模块职责或依赖方向。
- 引入新技术栈、框架或关键依赖。
- 改变公共接口、配置方式或运行方式。
- 改变界面风格、设计 token、可复用 UI 模式或 UI 协作流程。
- 改变模型供应商、语音服务、记忆策略或安全边界。
- 发现规则缺口或重复问题。

## 完成定义

一次任务只有在以下条件满足时才算完成：

- 修改范围与需求匹配。
- 未破坏已声明模块边界。
- 相关验证已执行或明确说明无法执行。
- 已评估运行态刷新需求。
- 风险、剩余问题和文档更新需求已在最终报告中说明。
